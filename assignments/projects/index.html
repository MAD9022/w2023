# 1. PWA - Giftr App

An installable, Offline-first PWA that lets the user create a list of people and their birthdays and then save gift ideas for each person. The data for each person gets saved in a Cache as a separate json file. That way it would be possible to export the person data. The App will work when offline or online.

The App will have TWO caches - one for the app files and one for the person .json files.

# Single Source of Truth

When building ANY application that involves data from localStorage, indexedDB, or the Cache API, then you need to optimize your performance by creating and maintaining a variable or property in your app script that stores a local copy of the data.

In this application our data is stored in multiple cached JSON files.

If we had to fetch and open EVERY file each time we went to the person list screen, it would really slow down performance. So, we only want to do this ONCE, when the app first opens. The sequence for doing this are:

  1. openCache
  2. get FileNames Array
  3. Loop and Open each File
  4. Add Each File's JSON content to the Single Source of Truth Array

From this point on, AFTER a successful ADD, EDIT, or DELETE of any person file, update the Single Source of Truth array with the new person object (or remove the person object for a delete). You only need to add, edit or delete the single person object within the array after you initial load, but only do this if the cache operation was successfully completed.

# App Screens

The different screens in the app are all part of the same HTML file. This app is a SPA.

You can either use different <div>s that you hide and show or you could create template elements that you clone and replace the innerHTML of the <main> element with the content for the next screen.

The different screens are:

  1. Home screen - shows the list of people, sorted by their birthday.
  2. Add/Edit Person screen - screen with a form that is used to add or edit a person and their date of birth.
  3. Gift List screen - shows a list of gift ideas for the selected person plus their name above the list.
  4. Add Gift screen - screen with a form for adding new gift ideas for the selected person.

# Home Screen

The Person list should sort only by the month and day, not the year. The list of people should appear in the order that their birthday occurs during each year.

Each person item in the list needs to show the name, the month and day of birth, plus two buttons - one to edit the person and one to navigate to the gift list screen.

Each time this screen is shown, rebuild the list of people.

While on the home screen the back button should NOT appear.

While on the home screen the Add button SHOULD appear. Clicking it will take the user to the add/edit person screen. Before navigating to the add/edit person screen set the selected person variable/property to ZERO or null.

Clicking on either the edit or gift button should save the id of the selected person in a variable/property to be used in the next screen.

# Add/Edit Person Screen

On the Add/Edit screen the way you know if you are adding or editing is with the value of the selected person variable / property. If it is ZERO or null, then you are adding a new person. If it is any value other than ZERO or null, then you are editing a specific person and you need to retrieve that file from the cache.

Use a temporary object that starts with the current information for a person that you are editing. Use a temporary object with properties that are empty for a new person. Eg:

//a new temp person for adding a person
let person = {
  id: crypto.randomUUID(),
  name: '',
  dob: 0,
  gifts: [],
};

To save a person, whether new or edited, step one is to pass the new temp person object as a File wrapped in a Response object to the cache. For an add you will need to generate a new person id. For an edit, use the existing previous id value. After a successful cache.put() to save the new file (overwriting a file of the same name), then update the Single Source of Truth with your temporary object.

The Export button in the video is an OPTIONAL feature. If you want to do this you can take the current data for the selected person, create a new File object and pass that to the URL.createObjectURL method. Then use that created URL as the href value for a dynamically created anchor tag and call the click() method on that anchor.

//optional export feature
let file = new File([JSON.stringify(person)], person.filename, { type: 'application/json' });
let a = document.createElement('a');
a.href = URL.createObjectURL(file);
a.download = person.filename;
a.click();

To delete a person, means deleting the file from the cache. AFTER the cache delete is successful, delete the person object from the Single Source of Truth, then navigate back to the Person List page, then rebuild the list of people.

After any successful add, edit, or delete person operation the app will navigate back to the person list (home) screen.

# Gift List Screen

The Gift List screen will use the selected person id to retrieve one of the person objects from the CACHE. This will give you the most recent data to update your Single Source of Truth.

It is possible that a Person was updated in another tab or window. We want the most up to date information in our Single Source of Truth before we update it ourselves.

As you add or delete gift ideas for this person in the interface you need to update the file in the cache with a new value.

Each list item in the gift list needs to display the gift idea text, the store, and url for the idea as a single column. Each list item also has a delete button to remove an idea from the list. Clicking the button should use a confirm() prompt before performing the actual delete. The delete operation means updating the single source of truth and then updating the person file in the cache, and finally updating the list on the page if the cache update was successful.

# Add Gift Screen

This screen is for adding new gift ideas only. No edit functionality is needed. It can optionally have a delete button. Deleting a gift from here means you need to navigate back to the gift list screen after a successful delete.

After saving a new Gift idea in the person .json file, navigate back to the gift list screen.

While on the Add Gift screen, the back button in the header SHOULD appear in the header and take the user back to the Gift list for the selected person.

While on the Add Gift screen, the Add button in the header should NOT appear.

# Header

The <header> element stays the same for every screen. It has three parts - a centered title, a back arrow, and an add icon.

The back arrow and the add icon will appear and disappear from the header as the user navigates through the app. The functionality of each will change depending on which screen is currently shown.

  • Home screen: No back button shown. Add button navigates to Add/Edit Person screen.
  • Gift List screen: Back button navigates to home screen. Add button navigates to Add Gift screen.
  • Add/Edit Person screen: Back button navigates to home screen. No add button shown.
  • Add Gift screen: Back button navigates to Gift List screen. No add button shown.

# Person Data Structure

Inside each Person .json file you need to store information about the person and about the gift ideas for them. Each file needs to have a unique name. The filename can also be kept inside the .json file, or you can use the person id as the file name (if using the crypto method to create the id). Just be sure to add the .json file extension for the filename.

Timestamps from Date.now() are one way to create a unique file name or id for a person or idea.

Another way is to use the crypto.randomUUID() method.

Remember that your file names need a file extension too.

When you initially create the person object, include the gifts property as an empty array.

Save the date of birth as a timestamp. When displaying the month and day you can extract that information from the timestamp by passing it to a new Date() object.

This project is a joint project between MAD9022 and MAD9124. The client-side React part of the project is for MAD9022 and the server-side MongoDB, NodeJS, Passport, and Express part is for MAD9124.

Make sure you let your instructors know who your partner is so the groups can be created in BS LMS for the assignment submissions.

# Project Summary

You are building a React-based version of the Giftr App. However, instead of saving the data as JSON files in the Cache you will be saving data in your API that you are building in the Server-Side course.

You will also need to add a Login page that communicates with your API to authenticate the user.

# Components and Layout

The React App is a Single Page App with routes to show different components.

  • login/logout
  • home (people list)
  • add/edit person
  • gift list for one person
  • add/edit gift idea

The login/logout route will show a login button if the user is not logged in and a message plus a logout button if the user is logged in. This is the "/" route.

The home (people list) will show a list of names and dates of birth for people that have been saved through the API. For each person show the name plus the month and day of birth. The list needs to be sorted by month, then day (the same way dates would appear on the calendar each year). If the list is empty, show a message about how to add a person. Each person item needs an edit button plus a button to navigate to the gift list.

The add/edit person component is used for adding a new person or editing an existing one. Pass in a user object as a prop to add or edit. For adding the id will be null, for editing it will actually have an id. The component will have an input for name and an input for the date. For both add and edit, show the Save button. When editing an existing user, a delete button should also be shown.

The gift list will show all the gifts for the selected person. The name of the person should appear on the screen too, with the list. If the list is empty, show a message about how to add a gift. Each gift item needs an edit button and a delete button.

The add/edit gift screen will have a three inputs: one for the gift idea; one for the store name; and one for the url. It will have a Save button too.

# Navigation

For this project, use the <BrowserRouter> from React Router to manage your navigation.

The login button will use the OAuth process to authenticate the user and return them to the same page, the "/" route. On successful authentication, save the JWT token in sessionStorage and then programmatically navigate to the person list route.

In the fixed header bar at the top of the screen, show a back button if the user is on the add/edit person route, the gift list route, or the add/edit gift route.

If the user is on the person list route then show a logout icon button instead.

On the person list or gift list route, in the header, show an Add icon button.

On each person item in the list there will be two icon buttons: one to go to the edit person route; one to go to the gift list route.

On each gift item in the list there will be one icon button to go to the edit gift route.

# Functionality

When entering a URL for a gift, check if the user wrote https:// at the start and add it if needed.

The login function work by using /auth/ endpoints in your API. The login will return a JWT string that you save in sessionStorage. To logout, you only need to delete the token from storage.

When a user clicks the login button, navigate to /auth/google?redirect_url=http://localhost:3000/login/success which will trigger the OAuth login and, when successful, will redirect to http://localhost:3000/login/success?token=eyt198nefo018e.1feef1ef1 with the jwt token in the querystring. You send the url route, that you want to use to handle the login, in the querystring with the param name redirect_url. The JWT string will be returned in the querystring with the param name token. The token string can be saved in sessionStorage and then added to any request headers with: Bearer: { Authorization: Bearer ${token}.

All the components that require someone to be logged in should have a common utility function that is called from inside a useEffect hook. If the user is NOT logged in (there is no token in sessionStorage), then you should programmatically redirect them back to the login route.

All of your fetch calls to your API should be made inside of a useEffect hook and they should all include the header Bearer: { Authorization: Bearer ${token} where the token is coming from sessionStorage.

A good way to handle the token in sessionStorage is to create a custom sessionStorage hook and put that inside a Context Provider (just like the Context API assignment).

Any delete of anything must have a confirmation dialog. It can be done with an HTML modal dialog, a UI Kit modal dialog, or the browser JavaScript native confirm() method.
