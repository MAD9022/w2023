# 1. Cache API

Starter code for this exercise will be provided during class.

This web app will let users enter values in an input, one at a time, to create a list. Then that list can be converted into a JSON File and saved locally with the Cache API. The starter code will handle the collecting of the information from the input and creation of the list display and saving of the values in an array.

The interface has a second button that will call a function to convert the array into a JSON File, wrap the JSON File in a Response Object and then save that Response locally with the Cache API. The filename for each file should include a timestamp (created with Date.now()) to ensure that each name is unique.

Once a Response is saved in the cache then that filename should be displayed on the page. When the page loads, the app should also check the Cache to see if there are any Response objects already there. If there are then display them as a list on the page.

The list of Files from the Cache need to be displayed as list items that show both the filename plus a delete button. Here is a sample of what each list item could look like.

<li>
  <span>data-162342934893.json</span>
  <button data-ref="data-162342934893.json" class="delete">Delete File</button>
</li>

If the user clicks on the <span> then that Response should be pulled from the Cache and the contents of the File displayed in the <pre><code> part of the webpage. The <span> should have a hover effect.

# 2. Service Workers

Starter code for this exercise will be provided during class.

For this assignment you will be given a basic webpage that has most of the HTML and CSS completed. You will need to add a couple web font links to the HTML and a few lines to the CSS. The JavaScript has the file structure and the code to make a fetch call which will create some data that will be used to add a list of users to the middle of the page.

In each of the files you will need to look for a series of comments with TODO: in each of the start files. These TODOs will provide you with a list of things that you need to complete.

The main JavaScript file will need to have the functions for detecting when the browser goes online and offline completed plus the function to register the service worker.

In the Service Worker file you will need to complete the event listeners for install, activate, and fetch. The sw.js service worker file also has TODOs and lists of instructions about what it needs to do.

The majority of the work for this exercise will be in the Service Worker file.

We want a service worker that will be able to retrieve and save files with the Cache API so that the page still works when offline.

The two main caching strategies that will be used are:

  1. For the html, css, js, fonts, and images we want to check the cache first and if the file is not there, then attempt to fetch it. CACHE FIRST
  2. For the JSON files we want to get the latest version by fetch. NETWORK FIRST If there is nothing in the cache then we could return a JSON file with an empty array or just let it fail. We can check if the browser is online or offline before attempting the fetch of the JSON file.

You can clone a copy of the starter code repository and then delete the .git folder. After doing that you can create your own Github private repo, run a git init locally and add the remote url. Alternatively, you can fork the Repository on Github and then clone your forked version.

# 3. React Routing

Hosting This Exercise

This exercise will be hosted on Netlify, NOT Github Pages.

This exercise will demonstrate your ability to create a basic React web application that has nested components, uses state variables with the useState and useEffect hooks, fetches data from the Random Data API (opens new window), and uses version 6 of the React Router library to control navigation to different screens in your application.

# Netlify

Your finished app will be hosted on Netlify... So, push to github early and set up Netlify to start displaying your Github repo's build folder early.

Remember to use the npm run build command in your Netlify setup so it rebuilds and hosts the build folder each time you push to Github.

These two videos will review how to use Netlify for hosting and how to connect your Github Repo to Netlify for automated updates.

# Layout

Your web app needs to have a <Header> component, a <Nav> component, and a <Main> component. The Header and Nav components are always visible.

The content inside the Main component will change depending on the Route. It could be the home page, or the list of items from an endpoint, or a list PLUS a details area, or a 404 screen.

When showing a list PLUS the details for one item from the list, the item in the list needs to be highlighted as the selected one. Using the React Router <NavLink> component is a good way to do this, as it has an isActive property that can be used to add a CSS className to the <a> that it will render.

# Data

The data for the application needs to come from the Random Data API (opens new window). Pick any two endpoints from the API. Eg: Banks and Beers.

Whichever endpoints you pick, retrieve between 20 and 30 items to display.

# Navigation

The <Nav> element needs to have a link displayed for the Home page (which is just a welcome page), and then one link for each of the API endpoints. Eg: HOME, USERS, APPLIANCES

Inside your <App> component you need to wrap <BrowserRouter> around the content for your app, inside the App() return statement.

The best place for your top level <Routes> and <Route> components is inside your <Main> component.

You can do your routing with or without the use of an <Outlet> component. (All the Routes inside Main.js or split between files.)

There needs to be a wildcard Route that handles all the invalid urls and sends the user to a 404-style page. This will be rendered inside <main>.

The <Home> component should be displayed for any of these four routes - /, /home, /index, /index.html

For each of your endpoints there needs to be two routes. One for the full list and one for a specific id. Eg: /users and /users/1234. When /users is the URL then you show the list of users. When /users/1234 is showing then you show BOTH the list and the details for user id 1234. The part that comes after /users/ will be interpreted as an id parameter.

# Components

Here is a list of the components that you should have JavaScript files and CSS files for.

  • <App>
    • <Header>
    • <Nav>
      • <NavItem>
    • <Main>
      • <Home>
      • <FourOhFour>
      • <Banks>, <Beers> (change these to match your data api endpoints)
        • <ListItem>
        • <Bank>, <Beer> (change these to match your data api endpoints)

Each level of nesting in the list above demonstrates the nesting of each component inside a parent component.

# Design

Create a custom colour scheme and use it throughout your application. Set up your colour variables inside your App.css file.

Use a DARK mode approach (darker background and lighter text) to styling your app.

The colours, font-sizes, and layout need to be responsive and accessible.

# 4. React Context

For this exercise we are combining React Router with the React Context API. You will have a private repository on Github with the code and connect it to Netlify so the hosted version will run on Netlify.

Use the React Router <HashRoute> for your navigation.

The app needs to have two routes / and /favs. The / route is the home page which will fetch data from an API and then display it as a list. Each item in that list will have a button that let's you toggle whether or not the item is saved in a favourites list. The /favs route will display each of the items that are currently favourited.

# Data Source

You can use ANY API that you want as the data for the App, as long as each fetch call to the API returns a new unique list of objects.

# Layout

The layout for your App should be responsive. The layout and the content should adjust to the available screen size. Adjust things like padding and font-size to help with fitting in the screen size.

# Home Route

Each initial render of the home page it should use a useEffect() call and fetch a new list of items.

When displaying each item, include a button with an icon for adding or removing the item from the favourites list.

The list of favourites will come from the Context Provider and should be accessed with the useContext() hook.

The home route will also use the useState() hook to create a state variable to hold the array of data to display in ListItem components.

You will need to create a toggle function that will add an item to the favourites list (from the Context) or remove an item from the favourites list (from the Context).

# Favs Route

The favourites page will use the useContext() hook to get the list of favourited items. That list will be displayed using an array of ListItem components.

No need for a button in these ListItem sub components.

# Context and Custom Hook

The list of favourites will be saved in LocalStorage. You can use the code for the custom localStorage hook in module 13.1 to achieve this.

Your Context API page will use the custom localStorage hook to send the list and its updates to LocalStorage. You can use the UserContext.js sample from module 13.1 as starter code but you will need to modify it to use the custom localStorage hook instead of useState().

# Component Structure

The basic component structure for the project will be something like this:

<App>
  <FavProvider>
    <header>
      <h1>Project title</h1>
    </header>
    <Routes>
      <Route path="/" element={<Items />} />
      <Route path="/favs" element={<Favs />} />
      <Route path="*" element={<Items />} />
    </Routes>
  </FavProvider>
</App>
